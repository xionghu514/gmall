# 购物车

* 购物车状态
    * 未登陆
        * 查询: 未登陆时查询未登陆的购物车记录
        * 新增: 如果购物车新增不包含某一个商品就会新增一条记录(新增)
        * 修改: 当购物车新增包含某一个商品 会对其数量进行累加(更新)
    * 已登陆
        * 查询: 先查询是否存在未登陆的购物车 如果存在 合并到 已登陆到购物车中, 并删除未登陆时购物车中的数据 展示已登陆的用户

## 技术选型分析

- 用户可以查询自己的购物车
- 用户可以在购物车中修改购买商品的数量。
- 用户可以在购物车中删除商品。
- 在购物车中展示商品优惠信息
- 提示购物车商品价格变化

### 存储方案

> 购物车是一个读多写多的场景, 可靠性要求不高. 每次对购物车操作就是对数据库做写操作选中 数量新增. 即使数据进行丢失 用户重新添加即可

* 浏览器端的存储技术: cookie(长度限制 每次请求都会携带) localStorage indexDB WebSQL
* 服务器端的存储数据: redis(性能高、成本也高, 不利于数据分析)、mysql(性能低、成本也低, 利于数据分析)
* 两者结合: redis(同步 性能 CRUD 主库) + mysql(异步 分析 辅库)

### 存储方式

* mysql: 单表
* Redis: Map<userId/user-key, Map<skuid, cartJson>>
    * 字符串: 类似于 java 中的 `Map<String, String>`
        * 以 userId 作为 key, 把 购物车的数据 `List<Cat>` 序列化为 json 作为 value
            * 可以实现只是性能十分低下, 用户每次对购物车进行操作时 我们都需要通过 userid 寻找到对应的数据 在反序列化成  List 对象进行遍历寻找到 对应的 sku 在对其做操作
    * 散列(hash): `Map<String, Map<String, String>>`
        * 首先不同用户应该有独立的购物车，因此购物车应该以用户的作为key来存储，Value是用户的所有购物车信息。这样看来基本的`k-v`结构就可以了。
        * 但是，我们对购物车中的商品进行增、删、改操作，基本都需要根据商品id进行判断，为了方便后期处理，我们的购物车也应该是`k-v`结构，key是商品id，value才是这个商品的购物车信息。
    * List: `Map<String, List<String>>`
    * Set: 类似于 `Map<String, Set<String>>`

## 功能实现

### 新增购物车

* 先获取登陆状态: 登陆 userid 未登陆 userkey
* 判断该用户的购物车是否包含该商品, 包含更新数量, 不包含新增记录

### 查询购物车

* 获取登陆状态: 登陆 userid 未登陆 uesrkey
* 未登陆则直接以 userkey 查询购物车并返回, 登陆则先合并未登陆的购物车, 清空未登陆的购物车, 最终返回合并后的购物车

1. 以 userKey 查询未登陆的购物车
2. 判断 userId 未空(判断是否登陆 userId == null). 如果未登陆则直接返回未登陆的购物车
3. 把未登陆的购物车 合并 到已登陆的购物车(userId)中去
4. 删除未登陆的购物车
5. 返回合并后的购物车

### 更新购物车

* 先获取登陆状态: 登陆 userid 未登陆 userkey
* 再根据 skuId 获取购物车对象, 更新购物车中数量即可

### 删除购物车

* 先获取登陆状态: 登陆 userid 未登陆 userkey
* 再根据 skuId 作为 key 删除对应的记录

### 总结

> 可以发现不管我们到哪一个步骤必不可免的都要查询用户的登陆状态, 可以提取拦截器中 统一获取登陆状态 登陆获取 userid 未登陆获取 userkey.

## 异步化改造

> 在日常开发中，我们的逻辑都是**同步调用**，顺序执行。在一些场景下，我们会希望异步调用，将和主线程关联度低的逻辑**异步调用**，以实现让主线程更快的执行完成，提升性能。考虑到异步调用的**可靠性**，我们一般会考虑引入分布式消息队列，例如说 RabbitMQ、RocketMQ、Kafka 等等。但是在一些时候，我们并不需要这么高的可靠性，可以使用**进程内**的队列或者线程池

* 本地异步: 一个工程内, jvm 4 种多线程方式。性能较高，可靠性相对较差
* 分布式异步: 跨工程, MQ异步。性能相对较低

* 三个问题
    * 获取异步任务的返回结果集(需要看到结果使用阻塞方式)
        * Future 对象: get() 阻塞方式
            * 方法修改为 public Future<返回结果> 方法名(),
        * ListenableFuture 对象: 监听非阻塞方式, 两个回调成功回调跟异常回调(不需要看结果 执行成功保存到数据库 某一天想看可以找到的时候可以使用非阻塞)
    * 获取异步任务的异常信息
        * Get 方法（阻塞）
        * 失败的回调（非阻塞）
        * 如果异步方法的返回值是非 future 对象：通过统一的异常处理器(非 Future 的对象)
            * 实现 AsyncUncaughtExceptionHandler 接口
            * 编写配置类实现 AsyncConfigurer 接口
    * 通过线程池控制线程数
